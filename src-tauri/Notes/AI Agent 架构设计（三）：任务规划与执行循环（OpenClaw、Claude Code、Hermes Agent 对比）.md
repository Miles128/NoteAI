# AI Agent 架构设计（三）：任务规划与执行循环

> **目标**：理解三个框架如何设计 Agent 的"思考-行动"循环，以及规划策略、执行策略、错误恢复背后的工程取舍
> **适合**：对 Agent 底层设计感兴趣，想真正理解"为什么这样设计"的读者
> **预计阅读**：15 分钟

## 任务规划：Agent 的核心难题

用户给 Agent 一个任务，Agent 需要把它拆解成可执行的步骤，然后逐步完成。这听起来简单，实际上是最难的部分。

难在哪里？

**第一，任务本身是模糊的。** "帮我重构这个模块"——重构到什么程度？要不要改接口？要不要加测试？用户不会把每一步都写清楚，Agent 需要自己判断。

**第二，执行过程中会遇到意外。** 第三步发现依赖冲突，原来的计划走不通了，需要调整。但调整到什么程度？是微调还是推翻重来？

**第三，长任务会遗忘。** 执行到第十步，Agent 可能已经忘了第一步的约束条件。它还在跑，但方向已经偏了。

三个框架对这三个问题的回答，揭示了三种根本不同的规划哲学。

## OpenClaw：ReAct 循环，模型自己规划

### 规划方式：隐式规划

OpenClaw 没有显式的"规划阶段"。用户给一个任务，Agent 直接开始执行——先想一步，做一步，看结果，再想下一步。这就是经典的 ReAct（Reasoning + Acting）循环。

每一轮循环，模型做三件事：

1. **观察**：看当前状态（上下文、工具返回结果）
2. **思考**：决定下一步做什么
3. **行动**：调用工具或生成文本

没有独立的"先规划再执行"阶段。规划和执行是交织在一起的。

### 优势：灵活，适应性强

隐式规划的最大优势是**灵活**。每一步都可以根据最新信息调整方向，不需要在开始时就做出完美计划。

遇到意外？下一轮循环自动调整。发现更好的方法？直接切换。用户中途改需求？模型立刻响应。

### 代价：缺乏全局视角

隐式规划的代价是**缺乏全局视角**。Agent 每次只看一步，容易陷入局部最优——在某个子问题上反复尝试，忘了整体目标。

更严重的是，长任务中 Agent 容易"跑偏"。没有一份明确的计划清单，执行到中途，Agent 可能已经偏离了用户的原始意图，但它自己不知道。

### 错误恢复：靠模型自己判断

工具调用失败时，错误信息作为 `tool_result` 注入上下文，模型自己决定：重试、换方法、还是向用户报告。

没有强制的重试策略，没有最大重试次数，没有自动降级机制。一切依赖模型的判断力。

## Claude Code：显式规划 + 结构化执行

### 规划方式：TodoWrite 工具

Claude Code 选择了完全不同的路径——**显式规划**。

任务开始时，模型用 TodoWrite 工具列出所有待办项。这不是写在回复里的文本，而是一个结构化的工具调用，每个 TODO 项有独立的 ID、状态（pending/in_progress/completed）和描述。

执行过程中，每完成一个步骤，模型更新 TODO 状态。每隔几轮，系统自动将当前 TODO 状态作为系统消息重新注入上下文——**不是提示词，是系统消息，比提示词更难被忽视。**

### 为什么选择显式规划？

Anthropic 在官方博客里坦诚解释了原因：

> 当我们最初推出 Claude Code，意识到模型需要一个待办列表来保持轨道。但即使这样，我们仍然经常看到 Claude 忘记它要做什么。为此，我们每隔 5 轮就注入系统提醒，提醒 Claude 它的目标。

这段话承认了两件事：一、目标漂移是真实存在的；二、解法不是靠模型更聪明，而是靠架构层面的提醒注入。

### 分层规划：从粗到细

Claude Code 的规划不是一次性的，而是分层的：

1. **任务级规划**：开始时列出所有主要步骤
2. **步骤级规划**：执行每个步骤时，再细化具体操作
3. **实时调整**：遇到意外时更新 TODO 列表

这种"从粗到细"的方式，兼顾了全局视角和执行灵活性。

### 错误恢复：结构化重试

Claude Code 的错误恢复比 OpenClaw 更结构化：

- **TodoWrite 状态追踪**：失败的任务标记为未完成，不会悄悄跳过
- **系统消息提醒**：定期提醒模型当前进度和待办事项
- **Task 工具**：可以派发子任务，子任务失败不影响主任务继续

## Hermes Agent：四阶段学习循环

### 规划方式：PLAN → EXECUTE → EVALUATE → LEARN

Hermes Agent 的规划是最结构化的，采用严格的四阶段循环：

**第一阶段：PLAN（规划）**

Agent 分析任务，生成执行计划。计划包含：目标分解、工具选择、预期结果。这个阶段不执行任何操作，只思考。

**第二阶段：EXECUTE（执行）**

按照计划逐步执行。每一步都有明确的预期输出，可以和实际输出对比。

**第三阶段：EVALUATE（评估）**

执行完成后，Agent 评估结果：是否达到预期？哪些步骤出了问题？哪些假设是错误的？

**第四阶段：LEARN（学习）**

从评估中提炼经验，更新 Agent 的知识库。如果这个任务值得沉淀，自动生成一份 Skill 文档。

### 迭代预算：硬性上限

Hermes 设置了迭代预算——默认 90 次工具调用上限。这不是建议，是硬性限制。超过上限，Agent 被强制停止。

这解决了一个实际问题：Agent 陷入死循环时，不会无限消耗资源。但代价是，复杂任务可能还没完成就被截断。

### 优势：经验积累

四阶段循环最大的优势是**经验积累**。每次任务完成后，Agent 不仅完成了任务，还学到了东西。下次遇到类似任务，可以直接复用之前的经验，不用从头摸索。

### 代价：开销大

四阶段循环的代价是**开销大**。每个任务都要经历完整的规划-执行-评估-学习流程，即使简单的任务也不例外。对于"帮我查个文件"这种简单请求，四阶段循环显得过重。

## 三种规划策略的对比

| 维度 | OpenClaw | Claude Code | Hermes Agent |
|------|----------|-------------|-------------|
| 规划方式 | 隐式（ReAct） | 显式（TodoWrite） | 结构化（四阶段循环） |
| 全局视角 | 弱 | 强 | 最强 |
| 灵活性 | 最强 | 强 | 中等 |
| 长任务稳定性 | 弱 | 强 | 最强 |
| 简单任务开销 | 最低 | 中等 | 最高 |
| 经验积累 | 无 | 无 | 有（自动生成 Skill） |
| 错误恢复 | 依赖模型判断 | 结构化重试 | 评估-学习循环 |

## 选择建议

没有哪种策略是绝对正确的，取决于你的场景：

- **任务简单、需要快速响应** → OpenClaw 的 ReAct 循环最轻量，反应最快
- **任务复杂、需要稳定执行** → Claude Code 的显式规划最可靠，不容易跑偏
- **任务重复、需要经验积累** → Hermes 的四阶段循环最合适，越用越聪明

最实际的做法是组合：简单任务用 ReAct 快速处理，复杂任务切换到显式规划模式，重复任务启用学习循环。三个框架都在向这个方向演进——根据任务复杂度自动选择规划策略，而不是一刀切。
