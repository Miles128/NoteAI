# AI Agent 架构设计（二）：工具系统设计

**目标**：理解三个框架如何设计 Agent 的工具系统，以及工具定义、权限模型、沙箱机制背后的工程取舍
**适合**：对 Agent 底层设计感兴趣，想真正理解"为什么这样设计"的读者
**预计阅读**：15 分钟

## 工具：Agent 的手脚

语言模型只能生成文本。它不能读文件、不能执行代码、不能访问网络、不能操作数据库。

**工具就是给 Agent 装上手和脚。** 没有工具的 Agent 是一个只会说话的大脑——什么都知道，什么都做不了。

但工具系统不只是"给模型加几个函数调用"这么简单。它涉及三个核心设计问题：

1. **工具怎么定义**——模型怎么知道有哪些工具可用，每个工具做什么，参数是什么
2. **权限怎么控制**——工具能做什么、不能做什么，边界在哪里
3. **安全怎么保障**——工具执行出错怎么办，恶意工具怎么防范

三个框架对这三个问题的答案，揭示了三种完全不同的工具哲学。

## OpenClaw：开放生态，社区驱动

### 工具来源：ClawHub 市场

OpenClaw 的工具主要来自 ClawHub——一个开放的技能和工具市场。任何人发布，任何人安装。规模庞大：超过 31,000 个 Skills 和工具，覆盖几乎所有工作场景。

这和 npm 生态的逻辑一样：**让最懂某个场景的人来写这个场景的工具。**

### 工具定义：MCP 协议

OpenClaw 采用 MCP（Model Context Protocol）作为工具的标准协议。每个工具通过 MCP Server 暴露，包含：

- **工具名称和描述**：告诉模型这个工具做什么
- **参数 Schema**：JSON Schema 格式，定义输入输出
- **执行逻辑**：实际的代码实现

MCP 的优势是标准化——所有工具遵循同一套协议，模型不需要为每个工具学习不同的调用方式。

### 权限模型：全权限

OpenClaw 的权限模型是**全权限**——安装的工具拥有和 OpenClaw 本身完全相同的系统权限。工具可以读写任何文件、执行任何命令、访问任何网络资源。

没有沙箱，没有权限分级，没有能力隔离。

这带来了严重的安全问题。2026 年 1 月的安全事件证明了这一点：ClawHub 上 341 个恶意工具植入了键盘记录器和信息窃取木马，恶意工具一度超过总量的 20%。

### 工具调用流程

1. 模型判断需要调用工具
2. 生成 MCP 格式的工具调用请求
3. OpenClaw 执行工具，返回结果
4. 模型根据结果继续推理

没有中间验证步骤，没有人工确认环节。模型决定调用，直接执行。

## Claude Code：分层工具，权限分级

### 工具来源：内置 + 自定义

Claude Code 的工具分两层：

**内置工具**：Anthropic 官方提供，包括文件读写（Read/Write）、命令执行（Bash）、搜索（Grep/Glob）、任务管理（TodoWrite/Task）等。这些工具经过严格测试，行为可预测。

**自定义工具**：用户通过 MCP Server 接入外部工具。和 OpenClaw 不同，Claude Code 的 MCP 工具默认权限受限，需要用户显式授权。

### 工具定义：三层结构

Claude Code 的工具定义比 OpenClaw 更精细，分三层：

1. **名称 + 描述**（始终加载）：告诉模型有哪些工具可用
2. **参数 Schema**（触发时加载）：定义工具的输入输出格式
3. **实现代码**（按需执行）：实际的执行逻辑

这种分层设计让模型在不需要调用工具时，只需要消耗极少 Token 就能知道"有哪些工具可用"。

### 权限模型：分级授权

Claude Code 的权限模型是**分级授权**：

- **只读工具**（Read、Grep、Glob）：默认允许，不需要确认
- **写入工具**（Write、Bash）：需要用户确认，或通过 `--allowedTools` 预授权
- **危险操作**（删除文件、安装包）：必须用户确认，不可预授权

Auto Mode 下，分类器会评估每个工具调用是否符合用户意图，不符合的自动拦截。这提供了一层独立于模型的安全保障。

### 沙箱机制：Bash 命令受限

Claude Code 执行 Bash 命令时，有一些内置限制：

- 某些危险命令（如 `rm -rf /`）会被拦截
- 网络访问受限于用户授权
- 文件操作受限于项目目录

但 Bash 本质上是一个万能工具，限制是软性的，不是硬隔离。

## Hermes Agent：最小权限，沙箱执行

### 工具来源：内置 + 插件

Hermes Agent 的工具也分两层，但和 Claude Code 的侧重点不同：

**内置工具**：精简到最少——只有文件读写、命令执行、网络请求等基础能力。Hermes 的理念是：工具越少，行为越可预测。

**插件工具**：通过 PLUG 插件系统接入。每个插件有独立的权限声明，用户安装时可以看到插件需要哪些权限。

### 工具定义：声明式权限

Hermes 的工具定义最特别的地方是**声明式权限**。每个工具在定义时必须声明：

- 需要访问哪些文件路径
- 需要执行哪些类型的命令
- 需要访问哪些网络地址

这些权限声明在安装时展示给用户，用户可以选择授权或拒绝。运行时，Hermes 严格按声明执行——工具试图访问未声明的资源，会被自动拦截。

### 权限模型：最小权限原则

Hermes 坚持最小权限原则：

- 工具只能访问它声明需要的资源
- 每个工具在独立的沙箱中执行
- 工具之间默认不能互相访问数据
- 敏感操作需要二次确认

这是三个框架里最严格的权限模型。代价是灵活性——某些需要跨工具协作的场景，配置起来比较繁琐。

### 沙箱机制：真正的隔离

Hermes 是三个框架里唯一实现了**真正沙箱隔离**的：

- 每个工具在独立的进程沙箱中运行
- 文件系统访问通过虚拟层重定向
- 网络访问通过代理层过滤
- 工具崩溃不会影响主进程

这种设计让 Hermes 在企业安全场景中有明显优势，但也带来了性能开销——沙箱的创建和销毁需要额外时间。

## 三种工具哲学的对比

| 维度 | OpenClaw | Claude Code | Hermes Agent |
|------|----------|-------------|-------------|
| 工具来源 | 社区市场（ClawHub） | 内置 + MCP | 内置 + 插件 |
| 工具数量 | 最多（31,000+） | 中等 | 最少 |
| 权限模型 | 全权限 | 分级授权 | 最小权限 |
| 沙箱机制 | 无 | 软限制 | 硬隔离 |
| 安全性 | 最低 | 中等 | 最高 |
| 灵活性 | 最高 | 中等 | 最低 |
| 冷启动体验 | 最好 | 中等 | 最慢 |

## 选择建议

- **追求快速上手、工具丰富** → OpenClaw，但要注意安全审查
- **平衡安全和灵活** → Claude Code，分级授权适合大多数场景
- **安全要求极高** → Hermes，沙箱隔离是企业级场景的刚需

工具系统的设计，本质上是在**灵活性**和**安全性**之间做取舍。OpenClaw 赌开放生态的规模效应，Claude Code 走实用主义的中间路线，Hermes 坚持安全优先的保守策略。没有绝对正确的选择，只有适合你场景的选择。
