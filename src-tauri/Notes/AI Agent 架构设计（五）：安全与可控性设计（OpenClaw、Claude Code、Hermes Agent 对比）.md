# AI Agent 架构设计（五）：安全与可控性设计

> **目标**：理解三个框架如何设计 Agent 的安全机制，以及权限控制、行为约束、审计追踪背后的工程取舍
> **适合**：对 Agent 底层设计感兴趣，想真正理解"为什么这样设计"的读者
> **预计阅读**：15 分钟

## Agent 的安全悖论

Agent 越能干，越危险。

这是一个悖论：你希望 Agent 能帮你做更多事，就需要给它更多权限。但权限越多，出错或被滥用的后果越严重。

具体来说，Agent 的安全风险来自三个层面：

**第一层：工具权限失控。** Agent 可以执行命令、读写文件、访问网络。如果权限控制不当，一个错误的工具调用就能造成不可逆的损害。

**第二层：提示注入攻击。** 恶意输入可以操纵 Agent 的行为——让它在执行任务时偷偷执行攻击者的指令。这是 Agent 特有的安全威胁，传统软件不存在这个问题。

**第三层：供应链攻击。** Agent 依赖外部工具和 Skills。如果这些第三方组件被植入恶意代码，Agent 就成了攻击者的跳板。

三个框架对这三层风险的应对策略，揭示了三种不同的安全哲学。

## OpenClaw：信任社区，事后补救

### 权限控制：全权限模型

OpenClaw 的安全模型最简单：**安装的工具拥有和 OpenClaw 本身完全相同的系统权限。**

没有权限分级，没有沙箱隔离，没有能力边界。工具能做什么，完全取决于工具的实现。

这和浏览器扩展的早期模型类似——安装一个扩展，它就能访问你所有的网页数据。Chrome 后来引入了权限声明机制，但 OpenClaw 至今没有类似的设计。

### 提示注入防护：依赖模型自身

OpenClaw 没有专门的提示注入防护机制。防御完全依赖模型自身的识别能力——模型需要自己判断哪些输入是恶意的，哪些指令不应该被执行。

这显然不够。研究表明，即使是当前最强的模型，面对精心构造的提示注入攻击，防御成功率也不到 60%。

### 供应链安全：事后审计

ClawHub 的安全策略是**事后审计**——任何人都可以发布工具，出了问题再处理。

2026 年 1 月的安全事件暴露了这种策略的致命缺陷：341 个恶意工具在 ClawHub 上存在了数周，影响了大量用户。事后审计发现恶意或有安全隐患的工具一度超过总量的 20%。

ClawHub 目前的发布门槛极低：只需要一个超过一周的 GitHub 账号。没有代码签名，没有沙箱验证，没有安全扫描。

### 用户能做什么？

OpenClaw 社区形成了一些安全实践共识：

- 在 `AGENTS.md` 中写入安全规则："禁止执行 `rm -rf`"、"禁止访问 `/etc` 目录"
- 安装工具前审查源代码
- 在隔离环境中测试新工具

**但这些都是软约束，依赖模型的遵从和用户的自觉。**

## Claude Code：分级授权，独立验证

### 权限控制：三级授权模型

Claude Code 的权限控制是三个框架里最精细的，分三级：

**第一级：默认允许（只读操作）**

- Read：读取文件
- Grep/Glob：搜索文件
- LS：列出目录

这些操作不会修改任何内容，风险最低，默认允许。

**第二级：需要确认（写入操作）**

- Write：写入文件
- Bash：执行命令
- Edit：编辑文件

这些操作会修改系统状态，需要用户确认。用户可以通过 `--allowedTools` 参数预授权特定工具，避免每次都确认。

**第三级：必须确认（危险操作）**

- 删除文件
- 安装系统包
- 修改配置文件

这些操作不可预授权，每次都必须用户确认。

### Auto Mode：独立视角的验证

Auto Mode 是 Claude Code 在安全设计上最有价值的创新。

Auto Mode 的分类器在每次工具调用执行前介入，评估这个调用是否符合用户的原始意图。关键设计是：**分类器看不到模型的推理过程，也看不到工具的执行结果——它只看用户的原始请求和即将发生的工具调用。**

这个"信息隔离"是刻意的防御。如果分类器能看到模型的推理，模型就可以在推理里解释"为什么这个操作是合理的"，从而说服分类器放行。隔离推理过程，意味着分类器只能判断：**这个工具调用，相对于用户的原始意图，合不合理？**

### 提示注入防护：系统消息隔离

Claude Code 对提示注入的防护主要靠**系统消息隔离**：

- 用户的原始指令作为系统消息注入，优先级最高
- 工具返回的结果标记为 `tool_result`，模型知道这些内容来自外部
- TodoWrite 状态作为系统消息定期刷新，防止被工具返回内容覆盖

这不是完美的防御，但比 OpenClaw 的"靠模型自己判断"强得多。

### 供应链安全：官方审核 + 社区审查

Claude Code 的 Skills 生态比 ClawHub 小得多，但安全门槛更高：

- Anthropic 官方提供的 Skills 经过内部安全审核
- 社区 Skills 需要用户手动安装到 `.claude/skills/` 目录，不会自动加载
- Skills 的加载采用渐进式披露，Agent 不会一次性执行未知代码

## Hermes Agent：最小权限，沙箱隔离

### 权限控制：声明式权限 + 沙箱

Hermes 的安全模型是三个框架里最严格的，核心原则是**最小权限**：

**声明式权限**：每个工具在定义时必须声明它需要访问的资源（文件路径、命令类型、网络地址）。运行时，Hermes 严格按声明执行——工具试图访问未声明的资源，会被自动拦截。

**沙箱隔离**：每个工具在独立的进程沙箱中运行。文件系统访问通过虚拟层重定向，网络访问通过代理层过滤，工具崩溃不会影响主进程。

### 提示注入防护：输入过滤 + 输出验证

Hermes 对提示注入的防护是双层的：

**输入过滤**：用户输入经过预处理，检测已知的注入模式（如"忽略之前的指令"、"你现在是..."）。匹配到的输入会被标记，模型收到额外的安全提醒。

**输出验证**：Agent 的输出在返回给用户前，经过安全检查。如果输出包含可疑内容（如执行了未声明的命令、访问了未授权的资源），会被拦截并报警。

### 供应链安全：权限声明 + 运行时验证

Hermes 的插件系统（PLUG）要求每个插件声明权限，安装时展示给用户。运行时，Hermes 验证插件的实际行为是否和声明一致。

如果插件试图执行未声明的操作，会被自动拦截，并记录安全日志。这是三个框架里唯一做到了**运行时行为验证**的。

### 迭代预算：防止失控

Hermes 设置了迭代预算（默认 90 次工具调用上限），作为最后一道防线。即使所有安全机制都被绕过，Agent 也无法无限消耗资源。

这解决的是**失控**问题——Agent 陷入循环或被劫持时，不会无限运行。但无法解决**欺骗**问题——Agent 在预算内完成了一个错误的任务，迭代预算检测不到。

## 三种安全哲学的对比

| 维度 | OpenClaw | Claude Code | Hermes Agent |
|------|----------|-------------|-------------|
| 权限模型 | 全权限 | 三级授权 | 最小权限 + 沙箱 |
| 提示注入防护 | 依赖模型 | 系统消息隔离 | 输入过滤 + 输出验证 |
| 供应链安全 | 事后审计 | 官方审核 + 手动安装 | 权限声明 + 运行时验证 |
| 失控保护 | 无 | Auto Mode 分类器 | 迭代预算 |
| 安全等级 | 最低 | 中等 | 最高 |
| 使用便利性 | 最高 | 中等 | 最低 |

## 核心洞察：安全和使用体验的权衡

三个框架的安全设计，揭示了一个根本性的权衡：**越安全，越不方便。**

- OpenClaw 的全权限模型让工具使用零摩擦，但代价是安全风险极高
- Claude Code 的分级授权在安全和便利之间找到了平衡点
- Hermes 的沙箱隔离提供了最强的安全保障，但配置繁琐，使用门槛高

这个权衡不是技术问题，是产品定位问题：

- 个人开发者、快速原型 → OpenClaw 的便利性更有价值
- 团队协作、生产环境 → Claude Code 的平衡策略最实用
- 企业合规、金融场景 → Hermes 的严格安全是刚需

未来的方向是**自适应安全**——根据任务的风险等级自动调整安全策略。低风险操作快速放行，高风险操作严格审查。三个框架都在朝这个方向演进，但目前还没有任何一个框架做到了真正的自适应安全。
